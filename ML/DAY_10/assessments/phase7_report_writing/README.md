# Phase 7 — Report Writing

**Goal:** Package everything into a professional EDA report — the deliverable decision-makers
actually read. This is the *final* phase, and the one your portfolio is judged on.

> 📄 **See the finished example:** [`../../EDA_demo_report/EDA_REPORT_DEMO.md`](../../EDA_demo_report/EDA_REPORT_DEMO.md). That file
> is the **quality standard** your own report should match.

## The Standard Report Structure (in order)

| # | Section | Purpose | Reader's question |
|---|---------|---------|-------------------|
| 1 | **Executive Summary** | Quick overview (3–5 sentences) | "What do I need to know?" |
| 2 | **Business Problem** | Context | "Why did we do this?" |
| 3 | **Dataset Description** | Transparency | "What data is this based on?" |
| 4 | **Data Cleaning Summary** | Trust | "Can I trust this data?" |
| 5 | **Key Findings** | Evidence | "What did we discover?" |
| 6 | **Visualizations** | Clarity | "Can I see it?" |
| 7 | **Recommendations** | Action | "What should we do?" |
| 8 | **Conclusion** | Closure | "What's the bottom line?" |

## Report Best Practices

- **Lead with the summary** — busy readers may read only that.
- **Be honest** — document cleaning and limitations; this *builds* trust.
- **Support every claim** with a number or a chart.
- **End with recommendations**, never with raw charts.
- **Use plain business language** — managers, not coders, read it.
- **Keep it scannable** — headings, bullets, charts.

> The report always ends with **recommendations and a conclusion**. The reader should finish
> knowing exactly what to do.
